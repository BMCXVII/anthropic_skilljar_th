Case Study: Agent
ผู้ช่วยที่มี goal + tools แล้วลงมือเอง

โจทย์: อยากได้ "ผู้ช่วยวิจัยตลาด" ที่รับเป้าหมายกว้าง ๆ แล้วตัดสินใจเองว่าจะค้นอะไร ใช้เครื่องมือไหน จนได้สรุป — นี่คืองานของ agent ไม่ใช่ workflow

🎯 แนวคิด + โครงสร้าง🧩 อิงคอร์ส Building with the Claude API📋 step-by-step
🧭
สถานการณ์ ความต่างหลัก: workflow = เรากำหนดขั้นตอนไว้ล่วงหน้า (คาดเดาได้/คุมง่าย), agent = ให้ goal + tools แล้วปล่อยให้ Claude คิดและลงมือเอง (ยืดหยุ่นแต่แลกกับ reliability/cost) เคสนี้เหมาะกับ agent เพราะขั้นตอนไม่ตายตัว
1

ถามตัวเองก่อน: agent หรือ workflow?

  • ถ้าขั้นตอน รู้แน่นอน (เช่น แปลไฟล์ → สรุป → ส่งอีเมล) → ใช้ workflow (chaining/routing/parallelization)
  • ถ้าขั้นตอน ขึ้นกับสิ่งที่เจอระหว่างทาง (วิจัยเรื่องที่ยังไม่รู้ขอบเขต) → ใช้ agent
2

กำหนด goal ให้ชัด

agent ทำงานจากเป้าหมาย ไม่ใช่ขั้นตอน — เป้าต้องชัดและ "ตรวจวัดได้ว่าเสร็จ"

เป้าหมาย: ทำสรุปคู่แข่ง 3 รายในตลาด X
ครอบคลุม จุดเด่น/ราคา/กลุ่มลูกค้า พร้อมแหล่งอ้างอิง
จบเมื่อได้ตารางเปรียบเทียบ + ข้อสังเกต 5 ข้อ
3

ให้ "ชุดเครื่องมือ" ที่จำเป็น

agent บรรลุเป้าด้วยการ เลือกใช้ tools เอง — ให้เท่าที่งานต้องการ เช่น web search, อ่าน/เขียนไฟล์, เรียก API ภายใน (ผ่าน MCP)

tools = [web_search, read_file, write_file]   # ให้เท่าที่จำเป็น
4

ปล่อยให้วน agent loop จนจบ

agent จะวน: ประเมินสถานการณ์ → เลือก tool → ดูผล → ตัดสินใจขั้นต่อไป จนบรรลุเป้า คุณสมบัติที่ทำให้ agent ดี: ใช้ tool อย่างมีโฟกัส, รับรู้สภาพแวดล้อม, ทำซ้ำแบบ iterative

✅
ได้ตารางเปรียบเทียบคู่แข่ง โดย agent ตัดสินใจเองว่าจะค้นคำไหน อ่านอะไร และเมื่อไหร่พอ
5

คุมความเสี่ยง

เพราะ agent ยืดหยุ่นและคุมยากกว่า workflow ให้: จำกัด tool/สิทธิ์, ใส่เงื่อนไขหยุด (เช่น จำนวนรอบสูงสุด), และให้ มนุษย์อนุมัติ ก่อน action ที่กระทบจริง

🎯 สิ่งที่ได้จากเคสนี้

  • agent = goal + tools แล้วคิด/ลงมือเอง; workflow = ขั้นตอนกำหนดไว้ล่วงหน้า
  • เลือก agent เมื่อขั้นตอนขึ้นกับสิ่งที่เจอระหว่างทาง
  • agent ที่ดี: focused tool usage + environmental awareness + iterative execution
  • คุมความเสี่ยงด้วยการจำกัด tool, เงื่อนไขหยุด และให้มนุษย์อนุมัติ
